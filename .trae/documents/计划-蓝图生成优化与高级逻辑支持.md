## Summary

目标：优化本项目的“蓝图”（workflow_spec）生成能力，提高对不同业务需求的泛化能力，并在保持 **可导入 Dify 的 DSL YAML** 约束下，支持多项高级逻辑：多分支路由、并行/汇合、循环/迭代、失败兜底。

核心策略：采用“混合式生成”——LLM 产出候选蓝图 + 规则化拓扑规范化（补齐/重排/汇合/兜底）+ 严格校验 + 编译到 Dify DSL。

## Current State Analysis

### 外部能力约束（Dify 支持）

根据 Dify 文档，Iteration/Loop 为原生节点，且迭代支持并行与错误处理策略（继续/终止/移除失败项）；Loop 支持终止条件、最大循环次数与退出节点等语义。这为“可导入 Dify 且支持循环/迭代/并行/兜底”提供基础能力。

## Proposed Changes

本改造分为 4 个层：Schema/生成、规范化与校验、编译到 Dify、测试与回归。

### 1) 扩展蓝图 Schema（workflow_spec v2 → v2+）

文件：[app.py](file:///D:/Projects/Difyprojects/dify/app.py)

- 扩展 `SUPPORTED_STEP_TYPES`：
  - 新增：`iteration`, `loop`, `variable_aggregator`, `parameter_extract`（从占位升级为真实节点）
- 约定新增节点的 `step.config` 结构（示例字段，后续实现中会固化为校验规则）：
  - `parameter_extract`：
    - `schema`: 输出字段定义（支持 Array/Object/Primitive），用于把文本提取成结构化变量
    - `source`: 从哪个上游字段提取（默认上游 llm.text）
  - `iteration`（容器节点）：
    - `items_selector`: 数组变量选择器（例如 `["parameter_extract_1", "items"]`）
    - `mode`: `sequential|parallel`
    - `error_policy`: `terminate|continue|null_keep|drop_failed`（映射到 Dify 的错误策略）
    - `subflow`: `{steps: [...], edges: [...]}` 作为迭代体内部子工作流
    - `output_key`: 迭代输出变量名（数组）
  - `loop`（容器节点）：
    - `loop_vars`: 可持久化变量定义（含初始值）
    - `termination`: 终止条件表达式/最大循环次数
    - `subflow`: `{steps, edges}` 循环体子工作流
  - `variable_aggregator`：
    - `mappings`: 归一化映射列表（把不同分支的变量映射到统一输出）

说明：容器节点（iteration/loop）的拓扑在 spec 层通过 `subflow` 表达，从而避免顶层 edges 出现 cycle，同时保留 Dify 原生循环/迭代语义。

### 2) 强化“LLM 产草图 + 规则规范化”的拓扑编排能力

文件：[app.py](file:///D:/Projects/Difyprojects/dify/app.py)

新增/改造模块（以函数为单位，集中放在 v2 相关函数附近）：

- 2.1 改造 LLM 规划 prompt（提高泛化能力）
  - 改 `llm_structure_requirement_v2` 与 `llm_plan_workflow_candidate_v2` 的 system prompt：
    - 明确“高级逻辑触发条件”与“节点类型使用准则”（何时用 iteration/loop/aggregator）
    - 输出更严格的 JSON schema（尤其是 edges/branch/subflow 的结构）
    - 增加 1-2 个最小示例（示例不输出 markdown，作为 JSON 片段 embedded）
  - 目标：减少“靠 deterministic_fix 兜底”的概率，提高候选 spec 初始质量。

- 2.2 新增拓扑规范化器 `normalize_workflow_spec_v2(spec, requirement) -> (spec, warnings)`
  - 输入：LLM 输出 + requirement（含 capabilities/node_preferences）
  - 输出：结构稳定、语义更强的 spec
  - 规则（优先级从高到低）：
    1. 结构稳定：id 唯一、类型合法、edges 引用存在
    2. 强制主链可达：确保 start→(核心链)→answer 可达
    3. 多分支路由：
       - ifelse 的 `config.cases[]` 与 edges 的 `branch` 必须一一对应；缺失则补齐或降级 direct
       - 分支的“汇合”若存在多个分支输出被下游消费：自动插入 `variable_aggregator` 节点统一输出（避免下游随机取第一个上游）
    4. 并行/汇合：
       - 当存在同一 source 连接多个节点且下游需要统一输入时：插入 aggregator 或 template/llm 聚合节点
       - 统一 edges 排序，避免“第一条边”影响 template/answer 的不稳定
    5. 循环/迭代：
       - 当 requirement/capabilities 指示批处理、列表处理或长文分片：将“对 items 重复处理”的子链提升为 iteration 容器
       - 当 requirement 指示“反复改进/直到满足条件”：构建 loop 容器，内含 ifelse/退出节点语义
    6. 失败兜底：
       - tool/code 后若下游是关键生成节点：插入 ifelse 校验节点（例如检查结果为空/错误关键字），失败走 fallback llm（降级路径）

- 2.3 评分函数增强（让更“可执行/可维护”的拓扑胜出）
  - 在 `_score_candidate_v2` 中加入加/扣分项：
    - 奖励：显式分支汇合（aggregator）、显式 answer 上游、iteration/loop 的合理使用
    - 惩罚：多入边但无汇合语义、ifelse cases 与 branch 不一致、关键节点缺少兜底

落点：`analyze_with_llm_v2()` 在 deterministic_fix 之后、validate 之前调用 normalize（保证所有候选都在同一规范化尺度下评分/校验）。

### 3) 扩展 Dify DSL 编译器（支持高级节点）

文件：[app.py](file:///D:/Projects/Difyprojects/dify/app.py)

- 3.1 新增种子模板文件（按文档手写最小可用结构）

    - `dify_seed_loop.yml`

- 3.2 编译逻辑扩展

    - `variable_aggregator`：编译为 variable aggregator 节点，写入 mappings（变量选择器映射）
  - 更新 `_to_dify_node_type` mapping 与必要的 handle 规则（若新节点需要特殊 sourceHandle/targetHandle）。

- 3.3 校验增强
  - 扩展 `validate_compiled_dify_yaml`：
    - 对新增节点类型做“结构存在性校验”（必需字段存在）
    - 校验容器节点子图引用的 id 唯一、edges 合法

## Assumptions & Decisions

- 输出必须保持为 Dify 可导入 DSL YAML（已锁定）。
- 高级逻辑优先覆盖：多分支路由、并行/汇合、循环/迭代、失败兜底（已锁定）。
- 种子模板采用“按文档手写最小可用结构”，并提供后续用 Dify 导出 YAML 替换 seed 的路线（已锁定，但存在导入兼容风险）。
- 前端暂不做可视化连线编辑；蓝图增强主要发生在后端规划/规范化/编译层。前端仅通过现有节点编辑（prompt/config JSON）承载必要配置。

## Verification Steps

实现完成后按以下步骤验收：

1. 运行单测：覆盖 validate/compile 对新增节点类型的用例（pytest）。
2. 运行服务并走通核心流程：
   - /analyze 生成含分支/并行汇合/兜底的 spec
   - /generate 成功生成 YAML、可下载、`validate_compiled_dify_yaml` 通过
3. 用 `yaml.safe_load` 解析生成结果，检查新增节点的 type 与关键 data 字段存在。
4. （推荐）将生成的 YAML 导入 Dify 实例做人工验收；如导入失败，优先修正对应 dify_seed_* 模板字段结构。
